DOCKER COMPOSE
Fait le lien entre les differentes images creees par les Dockerfile
afin de faire une application multi containers
Permet par exemple de definir les dependances (wordpress a besoin de nginx...)
pour assurer un bon fonctionnement de l'application

Network bridge :    
    - Isolation : Chaque réseau bridge est isolé des autres réseaux, les conteneurs d'un 
réseau bridge ne peuvent pas communiquer avec les conteneurs d'autres réseaux bridge sauf si connexion explicite.
    - Communication entre les Conteneurs : Les conteneurs connectés au même réseau bridge peuvent se communiquer 
entre eux via leurs adresses IP internes.
    - Connexion à l'Hôte : Les conteneurs connectés à un réseau bridge peuvent également se connecter à l'hôte Docker.
    - Connexions vers l'extérieur : Les conteneurs connectés à un réseau bridge peuvent accéder aux ressources réseau
externes telles que l'Internet, en utilisant le mécanisme de translation d'adresses réseau (NAT) fourni par Docker.

Q: How Docker and docker compose work
A: DOCKER : Container -> images, container, dockerfile
image : executable package that contains everything to execute a a piece of software : the code, libraries, env, conf file...
container : a container is the instance of an image.
dockerfile : script containing a list of commands to assemble an image.

Dockerfile is the makefile of containers. It is used to define and run a multi-container docker applications.
Services : basically the representation of the container that will be created.
Volumes : persistante data generated by and used by docker containers

RESUME : Docker manages individual containers each running a single thing (app, service...) while
Docker-compose manages multi-container applications or services with the use of a network 
(to communicate between the containers, to use volumes of diff containers etc), of image

Think of a Docker image as a blueprint or a snapshot of a filesystem. It’s like a recipe for baking a cake. The recipe itself doesn’t change, and you can use it to bake as many cakes as you like.
Think of a Docker container as a running instance of a program based on the recipe (image). If the image is the cake recipe, the container is the actual cake made from that recipe. You can eat, modify, or even discard the cake, but the recipe remains unchanged and can be used to make more cakes.

Docker Image:
    Think of a Docker image as a master copy of a book. It contains the complete content of the book, including all the chapters, illustrations, and formatting.
    The book itself doesn't change, and you can create as many photocopies of it as you need.
Docker Container:
    A Docker container is like a photocopy of the book. It's an instance of the book that you can read, annotate, highlight, or even tear pages out of.
    Each photocopy can be used differently, but they all originate from the same master copy (the image).
